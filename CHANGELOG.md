### v1.0.0

**🔀 首次发布：按窗口追踪 token 用量并自动路由模型**

**1. 📊 按窗口追踪用量**
* 以 UMO（Unified Message Origin）为单位记录每个对话窗口的 token 用量，不同窗口彼此独立计数。
* 在 `on_llm_response` 钩子中累计 `usage.total`（含输入与输出 token）。
* 用量数据持久化到插件数据目录的 `usage_data.json`，重启不丢失当日计数。

**2. 🔀 限额触发路由**
* 每个窗口可配置一条有序的模型路由链：模型 1 → 模型 2 → 模型 3 ……
* 当某模型当日用量达到 `daily_limit` 后，下一次请求自动切换到路由链中的下一个模型。
* 切换时机为「达到限额后的下一次请求」，当次请求仍由原模型完成。

**3. 🪂 用尽回退默认模型**
* 当路由链中所有模型都达到当日限额时，把该窗口回退到框架默认模型。
* 标记当天已用尽，当天不再重复处理该窗口，避免反复切换。

**4. 🕛 每日自动重置**
* 用量计数按本地时间每天 0 点重置。
* 通过日期字符串比较实现：每次记录用量时检查记录日期，若与今天不符则归零。
* 次日 0 点自动清除「已用尽」标记并恢复路由链。

**5. 🧷 Provider ID + 模型名双重标识**
* 每个模型条目同时包含 `provider_id` 与 `model_name`。
* `provider_id` 用于切换 Provider，`model_name` 用于覆盖请求的 `model` 字段。
* 防止不同提供商提供同名模型时产生混淆。

**6. 🧱 嵌套配置结构**
* 采用外层 `template_list`（窗口列表）嵌套内层 `template_list`（模型列表）的配置结构。
* 窗口数量与每个窗口内的模型数量均可自由添加，无固定上限。
* 每个窗口条目从上到下依次为：UMO 号、模型 1、模型 2、模型 3 ……

**7. 🤫 静默运行**
* 不提供任何命令，不发送任何提醒。
* 仅在后台完成用量统计与模型切换，切换行为通过 INFO 日志记录。
